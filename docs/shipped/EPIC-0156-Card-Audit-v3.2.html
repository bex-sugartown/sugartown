00Card Aesthetic & Intent
The aesthetic (from EPIC-0151)
Modern UI with library catalog structure cues. Think metadata discipline, monospaced labels, systematic layout — not paper, not tape, not skeuomorphic cosplay.

The Layout Contract

This is the intent behind the technical invariants in Section 04. State it plainly so no implementer rationalises their way around it:

  • → Header is pinned to the top — never shifts down, regardless of body length
  • → Footer is pinned to the bottom — never floats up, never absent when content is short
  • → Body expands and contracts between them based on content — it does the work, not the card
  • → Card works in a grid without fixed heights or brittle assumptions — no min-height, no percentage tricks, no magic numbers

These are the reasons __header and __footer get flex-shrink:0 and __body gets flex:1 1 auto. The CSS is the contract; the contract is the intent.

01Decisions Locked — Do Not Re-Open These

These were open questions in earlier versions. They are resolved. Implement exactly as stated.

Category placement
Category renders as a plain text link — "Category: [linked label]" — in a dedicated __category element. Not a chip. Two stories show both positions (above title vs. below eyebrow/above title) for visual comparison. Final position locked after Storybook review — tracked in Backlog BL-06. The component supports both via a categoryPosition prop defaulting to 'below-eyebrow' (current legacy behaviour from screenshots).
Project colorway / accentColor
accentColor?: string accepts a raw hex string from project.colorHex. Component derives tints using color-mix(): left border strip, eyebrow color override, header wash. Absent prop = no style applied = default. Brand color picker dropdown in Sanity Studio is Backlog BL-01.
KPIs in metadata
KPI fields removed from variant="metadata". KPIs surface as: (1) a pinned article/node at top of project page, or (2) a kpiLink prop rendering as "KPIs: [View Dashboard →]" in the footer. No data, no bars, no numbers inside any card.
Thumbnail position
variant="default": thumbnail renders as full-width hero above the header (inside card boundary). variant="listing": thumbnail renders as fixed-width left-rail column. No thumbnailPosition prop — position is determined by variant. No thumbnailUrl prop = slot is absent, card fills gracefully.
Full-card-clickable pattern
A card with href and no other interactive children uses the CSS expanded hit-target pattern: root is position:relative, title link gets ::after { position:absolute; inset:0 }. Other interactive children get position:relative; z-index:1. Never a wrapping <a> around other links — invalid HTML and inaccessible.
⚠️ Image strategy — featuredImage is deprecated
Option A (bridge now): thumbnailUrl is derived from hero.media[0] or the first image in sections[] via GROQ projection in apps/web. Component receives a resolved URL string or nothing. A dedicated cardImage schema field is Backlog BL-02.
DEPRECATED: featuredImage
The featuredImage field exists on article and caseStudy schemas for legacy data continuity only. Do not reference it in any new GROQ query, component, adapter, story, or documentation. It has been wired into CaseStudyPage.jsx and ArticlePage.jsx historically — those references are legacy and will be unwired in a future cleanup epic. The canonical image source going forward is sections[] content. Schema field removal is blocked by data-loss risk; Studio hide task is tracked in BL-07.
Density naming
Renamed from "dense" to "compact". density="compact". Applies to all variants. Produces tighter padding AND smaller type scale — not padding-only.

03Final Props Interface
// packages/design-system/src/components/Card/Card.tsx

interface CardProps {

  // Layout
  variant?:          'default' | 'listing' | 'metadata'  // default: 'default'
  density?:          'default' | 'compact'                 // default: 'default'

  // Header
  title:             string
  eyebrow?:          string              // "Node . PROJ-001" format
  category?:         { label: string; href: string }   // plain text link, not a chip
  categoryPosition?: 'below-eyebrow' | 'above-title'  // default: 'below-eyebrow'
  subtitle?:         string              // default variant only
  status?:           'draft' | 'active' | 'archived' | 'evergreen'
                     // | 'implemented' | 'validated' | 'deprecated'

  // Body
  excerpt?:          string              // default + listing only
  project?:          { label: string; href?: string }  // listing variant — project attribution

  // Metadata grid (variant="metadata" only)
  metadata?:         Array<{ label: string; value: string }>

  // Chip rows (default + metadata)
  tags?:             Array<{ label: string; href?: string }>  // filled pink chips
  tools?:            Array<{ label: string; href?: string }>  // outlined seafoam chips

  // Footer
  date?:             string              // ISO - component formats display
  nextStep?:         string              // "Next Step: [text]" - from legacy, optional
  aiTool?:           string              // "Claude" | "ChatGPT" | "Gemini" | "Agentic Caucus"
  stamp?:            string              // "Reviewed" | "Implemented" | "Due" etc.
  kpiLink?:          { label: string; href: string }   // link only - no data

  // Media - position determined by variant
  // Source: hero.media[0] or sections[] — NOT featuredImage (deprecated)
  thumbnailUrl?:     string
  thumbnailAlt?:     string

  // Project colorway
  accentColor?:      string              // hex from project.colorHex

  // Linking
  href?:             string              // full-card-link via ::after pattern
  className?:        string
}

Listing variant with thumbnail: root switches to flex-direction:row. Thumbnail becomes fixed-width left column (--st-card-thumb-rail-width). All other elements stack in right column using same slot names. Without thumbnail: flex-direction:column normal stack.

Invariants — what cannot change

  • → Root: display:flex; flex-direction:column (row when listing+thumb)
  • → __header and __footer: flex-shrink:0 — never compress or float
  • → __body: flex:1 1 auto — expands to fill space
  • → DOM order is semantic order: header → body → footer. No exceptions.
  • → No min-height on card root or grid wrapper
  • → No inline styles — all appearance via tokens and CSS Modules
  • → .st-card class always present on root
  • → No CMS types, GROQ, or Sanity shapes inside the component
  • → thumbnailUrl source is never featuredImage — use hero.media[0] or sections[]

These invariants encode the Layout Contract stated in Section 00. They are not styling preferences — they are the contract. If a future change would require overriding flex-shrink on header or footer, or adding min-height to the card root, the correct response is to re-open the brief, not to patch the component.

09Acceptance Criteria — Done Means All of These
  • Header and footer lock at 1-line and 10-line body content in a grid
  • Body expands without breaking card grid boundaries
  • Metadata grid label/value columns align at all label lengths
  • Default, Listing, and Metadata variants are visually distinct — not just padding differences
  • Compact modifier: tighter padding AND smaller type scale on all 3 variants
  • Light and dark themes render correctly with zero prop change
  • accentColor: left strip + eyebrow tint + header wash present. Absent prop = no style change.
  • Category renders as plain text link with label prefix, not as a chip
  • categoryPosition prop controls placement correctly in both modes
  • Listing with href + no other links: full card keyboard-navigable; ::after hit target pattern used
  • Listing without thumbnail: vertical layout. With thumbnail: left-rail column layout.
  • Default without thumbnail: no hero slot. With thumbnail: full-width hero above header.
  • kpiLink: renders as a plain link label only. No data, no bars, no numbers.
  • No min-height on card root or grid wrapper
  • Zero unresolved CSS custom properties in browser inspector
  • Zero hardcoded hex values or pixel values in component CSS
  • Zero CMS-specific types, imports, or logic in the component
  • No reference to featuredImage anywhere in component code, stories, or GROQ adapters produced by this epic
  • TypeScript compiles clean. Storybook builds clean.
  • CategoryBelowEyebrow and CategoryAboveTitle stories exist for visual comparison
  • Component re-exports correctly from design-system/src/index.ts
  • Legacy aliases (.pink-card, .gem-card) documented as deprecated in README with migration path
  • README includes featuredImage deprecation notice in image strategy section
  • README complete: purpose, decisions, anatomy, props, tokens, accessibility, non-goals, backlog

11Explicit Non-Goals
  • ✗ No skeuomorphism — not paper, not tape, not catalog texture. Information discipline as modern UI.
  • ✗ No CMS shapes inside the component — no Sanity types, no GROQ, no _type checks.
  • ✗ No KPI data rendering on any card — kpiLink is a link only.
  • ✗ No page-specific CSS — must work on any surface.
  • ✗ No WordPress dependencies.
  • ✗ No parallel card component classes — extend props, don't create siblings.
  • ✗ No fixed heights — the grid decides, the card fills.
  • ✗ No styled-components, Tailwind, or global class pollution. CSS Modules only.
  • ✗ No wrapping <a> tag around cards with child links — use ::after hit target pattern.
  • ✗ No thumbnail position prop — position is determined by variant, period.
  • ✗ No reference to featuredImage in any new code, query, story, or documentation produced by this epic.
