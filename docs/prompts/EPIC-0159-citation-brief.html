EPIC-0159 st-citation — Knowledge Notation System packages/design-system · Pink Moon era Created 2026-03-05 Typography Layer P1 · Non-blocking
Origin — Legacy Proof of Concept
This component was not invented here. The citation pattern — inline superscript [n] + monospace footnote caption — was already shipping in the legacy WordPress card layout, as visible in the pre-migration archive UI. This epic formalises what was discovered, not designed. The goal is extraction, not invention.
Overview
What
A formally specified st-citation component — consisting of an inline superscript marker and a footnote caption zone — extracted from the legacy card layout and codified into the design system typography layer with named tokens.
Why
Citations are what make Sugartown a knowledge system rather than a blog. They are the single most distinctive UI pattern in the site. Without a formal component contract, the pattern drifts, duplicates, and loses its scholarly intentionality across content types.
Layer
Typography layer — not the card layer. Cards accommodate citations; they do not own them. The component must render correctly inside st-card__body, inside long-form article prose, and anywhere body copy appears.
Scope
packages/design-system — tokens, CSS, React component, Storybook stories. No Sanity schema changes. No new routes. No hover behaviour or JS interaction in this epic.
Non-Goals — Explicit Scope Exclusions
Does not include hover preview cards or JavaScript tooltip behaviour. That is st-definition-card — a separate future epic.
Does not include a full bibliography page or st-citation-list organism. Tokens are specced here; the organism ships in a later epic.
Does not wire citations to live Sanity node data. This epic delivers the visual primitive only. CMS wiring is an application concern.
Does not auto-number citations. Numbering is handled in content authoring (Sanity portable text), not the component.
Does not modify any existing card components. st-card is untouched. The citation component slots into st-card__body naturally as body copy.
Is not a cutover blocker. This is a P1 typography enrichment. It ships when Pink Moon bandwidth allows.
Visual Contract — Rendered Example

The component has two parts: an inline marker (st-citation__marker) that appears inline within body copy, and a footnote caption (st-citation__note) that appears in a dedicated footnote zone at the bottom of the content surface. The example below shows both parts in a light-mode card context — which is where the pattern was first observed in the legacy UI.

Note that the citation footnote renders in IBM Plex Mono at a reduced size with muted colour — the same typographic register used for metadata throughout the system. This is not a coincidence; it is a semantic signal that the footnote is reference material, not narrative content.

Token Specification — New Semantic Aliases

No new base primitives are required. All citation tokens are semantic aliases that point to existing base tokens. This is cheap to implement and keeps the token graph clean. Read existing token files before writing anything.

Token Maps to Purpose
--st-citation-color var(--st-color-muted) Footnote text colour — same register as metadata
--st-citation-index-color var(--st-color-pink) The [n] number in the footnote — anchors to brand
--st-citation-font var(--st-font-mono) IBM Plex Mono — scholarly apparatus register
--st-citation-size 0.72rem Smaller than body — clearly reference, not narrative
--st-citation-marker-bg var(--st-color-pink) Inline marker pill background (light mode)
--st-citation-marker-color #ffffff Inline marker text — white on pink
--st-citation-zone-border rgba(255,36,125,0.2) Dashed divider above footnote zone
--st-citation-zone-gap var(--st-space-3) Top padding on footnote zone

Storybook Spec — Required Stories
①
Default — Single citation in card body
Shows CitationMarker inline within body copy, with a single CitationNote in a CitationZone below. Uses light mode card context. This is the canonical, approved usage.
②
Multiple — Three citations in one zone
Three inline markers, three corresponding notes. Validates that the zone stacks notes correctly and the index numbers align. Use the legacy CV / Resume card as the content reference.
③
In-prose — Citation inside long-form article body
Shows the component outside a card context, embedded inside a block of article copy. Validates that CitationZone is not card-dependent — it can follow any prose block.
④
Stress — Long footnote text
A citation note that wraps to two lines. Validates that multi-line notes don't break alignment of the [n] index with the note text.
⑤
Token audit — Documented tokens
A static story that renders all eight --st-citation-* token names alongside their current resolved values. Serves as a living reference in the DS documentation.
