EPIC-0147 Schema ERD — Static Page in apps/web apps/web · P1 Created 2026-02-28 Non-blocking
Overview
What
Add a live, interactive Schema ERD page to apps/web at route /schema-erd. The ERD component and its data manifest are committed to the monorepo. No external data fetching at runtime.
Why
Demonstrates MACH architecture literacy in the portfolio. Gives Bex and future collaborators a navigable map of the content model without opening Sanity Studio. Useful for onboarding, documentation, and press.
Surface
apps/web — React/Vite frontend only. No Sanity schema changes. No Studio changes. No new packages.
Upgrade path
Static manifest today → swappable with a GROQ-fetched or build-time generated manifest in the future. The component is data-source agnostic by design.
Non-Goals (Explicit Scope Exclusions)
Does not read live Sanity schema files or TypeScript source at runtime. The ERD data is a maintained JS constant in the repo.
Does not require a build-time codegen step. That is Option A — a future upgrade.
Does not modify any Sanity Studio schemas, GROQ queries, or apps/studio files.
Does not require a new package in packages/. This is app-local, not shared.
Is not a P0 cutover blocker. Backlog priority: P1. Ships when other P1 items allow bandwidth.

User Stories
ERD-001 Static manifest file P1
As a developer, I want a committed schemaManifest.js in apps/web/src/data/ so that the ERD has a canonical, version-controlled data source that is easy to update when the schema changes.
Acceptance Criteria
  • File exports two named arrays: entities and relationships
  • All 35 Sanity schema types are represented across four groups: atoms, taxonomy, content, infrastructure
  • Each entity includes: id, label, kind (object|document), group, color, fields[]
  • Each relationship includes: from, to, label, type (one|many|self)
  • File has a comment header: // Last synced: YYYY-MM-DD — update when schema changes
ERD-002 ERD component — props-driven P1
As a developer, I want SchemaERD.jsx to accept data as props rather than importing directly from the manifest, so that the component is reusable, testable, and ready for a future dynamic data source.
Acceptance Criteria
  • Component signature: <SchemaERD entities={[]} relationships={[]} />
  • No direct imports of schemaManifest.js inside the component file
  • Group filter tabs render from entities data — not hardcoded strings
  • Clicking an entity highlights it and shows its relationships in the sidebar
  • Clicking a related entity in the sidebar navigates to that entity
  • Clear selection button resets state
ERD-003 Route at /schema-erd P1
As a visitor, I want to navigate to /schema-erd and see the interactive content model diagram so I can understand the Sugartown CMS architecture without access to Sanity Studio.
Acceptance Criteria
  • Route /schema-erd is registered in the app router
  • Page renders without Sanity API calls — no loading spinner on first paint
  • Page <title> is "Schema ERD · Sugartown"
  • Page includes a meta description suitable for SEO
  • No console errors on mount
ERD-004 Design system tokens applied P1
As Bex, I want the ERD page to use Sugartown design tokens so it looks native to the site and doesn't break the visual system.
Acceptance Criteria
  • All colors reference st-* CSS tokens — no hardcoded hex values in the component CSS
  • Typography uses --st-font-mono and --st-font-sans variables
  • Background, surface, and border colors match the midnight dark theme palette
  • Passes visual inspection against existing site pages
ERD-005 Maintenance documentation P2
As a solo operator, I want clear instructions for updating the manifest when the schema changes so I don't have to rediscover the process 6 months from now.
Acceptance Criteria
  • Comment block at top of schemaManifest.js explains the data structure
  • A 3-5 line "How to update" note is added to the relevant section of the Sanity README
