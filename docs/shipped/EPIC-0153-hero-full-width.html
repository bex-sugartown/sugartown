What This Epic Does
Two workstreams, one epic This epic has two distinct but related bodies of work: (A) giving editors a choice of hero image display width, and (B) normalising content-width containers across all detail page types. Both are layout/CSS changes — no new schema document types.

Workstream A — Hero image width option. The heroSection schema gains a new imageWidth field with two values: full-width (edge-to-edge viewport banner) and content-width (fills the content area, respects the page's max-width column). Zero border-radius on all hero banners regardless of option chosen.

Workstream B — Content width normalisation. The current site has at least two conflicting content-width values in use: 800px (About page, via --st-content-width) and 1036–1100px (via --st-page-max). The canonical targets are 760px for detail/reading pages and 960px for archive/listing pages (archive application deferred). This epic defines the tokens, audits all containers, and applies the detail width consistently.

Workstream A — Hero Image Width Option

The heroSection schema object receives a single new field: imageWidth. This is an editorial choice per page — not a global default. The option controls how the hero banner renders relative to the page layout.

User Stories
0153-S1 Audit — Inventory all content-width containers Audit
Claude Code reads every page component and its CSS. Documents: which width tokens/values are in use, where --st-content-width and --st-page-max are referenced, actual rendered widths on each page type. Produces written inventory before any file is modified. Confirms canonical token names with Bex.
0153-S2 Define canonical width tokens CSS Tokens
Add --st-width-detail: 760px and --st-width-archive: 960px to the semantic token file. Add comment block: /* ── Content Width Scale ── */. Remove --st-content-width and --st-page-max after all references have been updated. No page components modified in this story.
0153-S3 Apply --st-width-detail to all in-scope detail pages Web · Pages
Update content container max-width on ArticlePage, CaseStudyPage, NodePage, and catch-all Page (incl. About) to max-width: var(--st-width-detail). Remove any references to old tokens. Confirm each page renders at 760px with appropriate horizontal padding on mobile. Visually verify About page at 760px — this is 40px narrower than its current 800px; flag if content feels cramped.
0153-S4 Add imageWidth field to heroSection schema Studio · Schema
Add imageWidth radio field to apps/studio/schemas/objects/heroSection.ts. Options: full-width | content-width. Default: content-width. Help text in Studio UI as specified. No other schema fields changed in this story.
0153-S5 Wire imageWidth to frontend hero rendering Web · Component
The heroSection renderer (in PageSections.jsx or the HeroSection component) reads imageWidth from the Sanity block and applies: full-width → full-viewport breakout class; content-width → max-width container class aligned with --st-width-detail. Both options: border-radius: 0.
0153-S6 Border-radius sweep — all hero contexts CSS · Sweep
Grep sweep for border-radius in hero-related CSS and borderRadius in JSX/TSX. Remove or zero-out any radius on hero containers, image wrappers, background overlays. Border-radius is permitted on elements inside the hero (chips, buttons, badges) — not on the hero band itself.
0153-S7 Confirm no radius-control field in heroSection schema Schema · Audit
Audit heroSection schema for any field named borderRadius, rounded, or similar. If found: remove (if unused) or apply hidden: true and add a comment explaining why. Editors must not be able to re-enable rounding on heroes.
0153-S8 Responsive — all widths hold at mobile / tablet / desktop Layout · QA
Verify at 320px, 768px, 1280px: full-width hero has no horizontal scroll; content-width hero aligns with the content column below it; 760px content columns have enough horizontal padding that text doesn't touch the viewport edge on mobile.

Archive Page — Width & Hero Treatment

Why deferred: Archive pages have a more complex layout (potential filter column, card grid, pagination) that warrants its own scoping pass. Mixing archive layout work into this epic risks scope creep and regression in the filter/grid system.

What was decided in this epic: The canonical token --st-width-archive: 960px is defined and ready to use. The ArchivePage component is untouched — it retains whatever max-width it currently has until the archive polish epic picks this up.

What the future epic needs to do: Apply --st-width-archive to the ArchivePage content container. Decide on archive hero treatment — the imageWidth field added to heroSection in this epic was not added to archivePage's hero field. That is an archive epic concern. Verify the filter column layout is compatible with 960px.

Page Templates — Not Active, Not Scoped
Status: Deferred indefinitely — open questions unresolved The concept of editor-selectable page layout templates was raised during scoping. It is explicitly not in scope and is parked here pending resolution of the questions below. Do not add a pageTemplate field to the page schema until at least one concrete non-default use case exists and has been designed.
Proposed TemplateIntentOpen Question / Concern
Default (Standard Width) Narrow reading column — maps to --st-width-detail (760px) This is already the normalised default applied in this epic. Is there a genuine need for this to be a selectable template if all detail pages default to it? Likely redundant as a picker option — it is simply the base behaviour.
Full Width Named "full width" but this does not mean edge-to-edge — that is the hero image option. "Full width" here means "wider than default" — i.e. a page whose content container uses --st-width-archive (960px) or larger. What is the concrete use case for a detail page (not an archive) that needs a wider content area? If it is primarily for card grids or gallery layouts, a section-level width override may be cleaner than a page-level template. Needs a real content example before scoping.
With Sidebar Two-column layout with a sidebar What goes in the sidebar? No current content types or design patterns include a sidebar concept. This requires a design decision — related content? table of contents? navigation? — before any technical work. Do not build a sidebar without a defined, designed use case.

Recommendation: The current implementation — all detail pages at 760px, controlled by a CSS token — is cleaner than exposing an unmotivated template picker to editors. Revisit page templates when a specific layout need arises that cannot be served by the existing section builder.
