EPIC-0161

Card Grid & Typography Polish

Priority · NEXT Design System · UX 3 workstreams · ~1 sprint
Context & Motivation

Three related visual polish issues have been accumulating since EPIC-0158 shipped the card adapter migration. They share a single surface — the archive card grid — and a single root cause: the DS Card component was built to spec for full-width contexts (homepage, case study pages) but hasn't been tuned for the narrowed viewport created when the FilterBar sidebar is present.

These are bundled into one epic because all three changes touch packages/design-system/src/components/Card/, and shipping them separately would generate three near-identical PRs to the same files.

Affected surfaces
Archive page (all doctypes) · Any future page using CardBuilderSection with FilterBar
Dependency
EPIC-0158 (card adapter) must be merged. EPIC-0160 (CardBuilderSection) may run in parallel — share token decisions.
Layers touched
packages/design-system · apps/web · apps/storybook
Hard constraints
No new CSS class names without checking existing classes first. Token changes must update DS token file + web token file in the same commit.
Workstream A Card Grid Layout — FilterBar Narrow Context
Problem: When FilterBar is present, the grid container width drops to ~600–700px. auto-fill minmax(300px, 1fr) produces 2 narrow columns that squeeze cards below their designed minimum. The card visually degrades into an unreadable sliver.
Implementation tasks
A1
Read before writing. Open packages/design-system/src/components/Card/Card.module.css and apps/web/src/components/ArchivePage/ArchivePage.module.css (or equivalent). List every existing grid / layout class. Do not proceed until this is done.
gate
A2
Enforce 1-column layout below 900px container width for archive contexts where FilterBar is active. Use a data-context="narrow" attribute on the grid wrapper (set by the archive page when FilterBar is mounted) rather than a breakpoint media query, so this doesn't accidentally fire on full-width grids at the same viewport width. Preferred approach: container queries (@container) if browser support is confirmed in the project. Fallback: [data-layout-context="narrow"] .st-card-grid { grid-template-columns: 1fr; }
layout
A3
Reserve 2–3 column grid for full-content-width contexts only (homepage, no sidebar). Document this constraint in a CSS comment above the grid rule.

The toggle should persist the user's choice in sessionStorage keyed to the archive slug — not a URL param. This is the "explicit user control" option from the backlog item. Scope to archive page only for now — do not generalise to a DS component.

Workstream B Card Title Typography — Scale & Font Switch
Problem: The card title renders at too large a weight and size for a dense catalog grid. It reads like a hero heading. In a 3-column grid with 6–12 cards, every title competes for dominance. The result is visual noise rather than scannable hierarchy.
Typography decisions (locked)
—
Card title: font-family: var(--st-font-sans) (DM Sans, already loaded globally) · font-size: ~1rem · font-weight: 600 IBM Plex Mono is reserved for eyebrows and metadata labels only. DM Sans at 600 weight is readable and compact without shouting.
decision

Workstream C Chip Row Group Labels — Tools · / Tags ·
Problem: In the card body, tool chips and tag chips render in the same visual style with no label to distinguish them. A user scanning the archive cannot tell whether a chip is a tool used or a topical tag without prior knowledge of the taxonomy.
Note: The CSS class .st-label already exists in the legacy stylesheet (style_260118.css). Verify whether it has been ported to the design system token/component layer before creating anything new. Do not duplicate.
Implementation tasks
C1
Read before writing. Search packages/design-system/src/ for any existing st-label, .label, or chip-group-label pattern. Search apps/web/src/ for the same. Record findings before writing any code.
gate
C2
In the DS Card component's chip-row rendering in the card body, prepend a <span className={styles.chipGroupLabel}> element before each chip group. The label text is passed as a new optional prop: toolsLabel?: string (default: 'Tools') and tagsLabel?: string (default: 'Tags'). The separator dot (·) should be rendered in CSS via ::after pseudo-element, not hard-coded in the label string. This keeps the label string clean for i18n.
ds
C3
Style the group label. If .st-label has been ported to the DS token layer, apply it. If not, create .chipGroupLabel in Card.module.css using tokens: font-family: var(--st-font-mono) · font-size: 0.65rem · color: var(--st-color-text-muted) · white-space: nowrap. Do not hardcode hex values.
ds
C4
Only render the label if the chip group is non-empty. If tools[] is empty, do not render the Tools label. Same for tags.
ux
C5
Add a Storybook story: CardChipLabels showing three states — tools only, tags only, both groups — to confirm labels appear and disappear correctly with empty data.

Rollback Plan

No Sanity schema changes in this epic. No migration scripts. No data risk. All changes are CSS/TSX within the DS and web app.
